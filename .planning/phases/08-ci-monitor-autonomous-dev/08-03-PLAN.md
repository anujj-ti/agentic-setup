Script requirements:
- Shebang: `#!/usr/bin/env zsh` + `set -euo pipefail`
- Positional args: `REPO="${1:?repo required (OWNER/REPO)}"`, `ISSUE_NUM="${2:?issue number required}"`
- Optional flag: parse `--dry-run` from args; if present, skip the `gh` call and return a canned JSON stub with `"dry_run": true` (used by verify-phase-08.sh for smoke testing without live network calls)
- GH binary: `GH="/opt/homebrew/bin/gh"` (explicit path, never rely on PATH)
- gh call: `$GH issue view "$ISSUE_NUM" -R "$REPO" --json title,body,labels,milestone,assignees,number,url`
- Use python3 for JSON parsing and output construction — do NOT use `jq` for multi-field extraction (fragile with nested objects)
- Body truncation: truncate `body` field to 2000 characters to prevent injection when the issue body is later used in Beads task titles (SECURITY.md rule)
- Error handling: if the `gh` call fails (non-zero exit), write `{"ok": false, "error": "gh issue view failed: <stderr>"}` to stdout and exit 1
- stdout: structured JSON matching the shape in the interfaces block above
- stderr: `echo "fetching issue #$ISSUE_NUM from $REPO..." >&2`
- Make executable: `chmod +x scripts/devbot-intake-issue.sh`

The `--dry-run` canned response should contain recognizable test data:
```json
{

  "repo": "OWNER/REPO",
  "dry_run": true,
  "issue": {
    "number": 0,

    "labels": ["test"],
    "milestone": null,
    "assignees": [],
    "url": "https://github.com/OWNER/REPO/issues/0"
  }
}
```

SOUL.md additions — add a new section "## Autonomous Development Workflow (DEV-04)":

The autonomous dev flow has two phases:

**Phase A — Issue Intake (when Task Orchestrator delegates an issue):**
1. Receive a `sessions_spawn` from Task Orchestrator with the repo and issue number
2. Run `exec scripts/devbot-intake-issue.sh OWNER/REPO ISSUE_NUM`
3. Return the structured JSON payload to Task Orchestrator (do NOT create the Beads epic)
4. Task Orchestrator creates the 5-subtask Beads epic and spawns you again

**Phase B — Beads Execution (when spawned with "Your tasks are in Beads. Run `bd ready --json` to start."):**
1. Run `BEADS_DIR="$HOME/.openclaw/beads" /opt/homebrew/opt/node@24/bin/bd ready --json`
2. Claim the first ready task: `bd update <task-id> --claim --json`
3. Execute the task (design, implement, self-review, QA evidence, or open PR depending on the task)
4. Close with factual evidence: `bd close <task-id> --reason "<what was done and what evidence exists>"`
5. Repeat from step 1 until `bd ready --json` returns an empty list
6. Report completion to Task Orchestrator

**CRITICAL RULES:**
- NEVER create Beads epics — only Task Orchestrator creates epics (ORCH-03)
- NEVER merge PRs — PR creation (T5) opens a draft PR only; merge is Phase 10
- NEVER give free-text instructions; wait for Beads graph to be pre-created before claiming
- Close reasons must be factual evidence strings: "Created PR #47 at https://github.com/..." not "I completed the task"

|----------|-------------|
| GitHub issue API → devbot-intake-issue.sh | Issue title/body/labels are untrusted; body is truncated to 2000 chars |
| devbot-intake-issue.sh → Task Orchestrator | Structured JSON output is trusted internally but body field is bounded |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-07 | Tampering | issue body → shell string | mitigate | Body extracted and truncated by python3 (2000 chars max); stored as a Python variable, never interpolated raw into a shell command |
| T-08-08 | Information Disclosure | gh token in devbot-intake-issue.sh | accept | `gh` CLI manages auth token via its own Keychain integration; token never appears in the script or stdout |
| T-08-09 | Elevation of Privilege | DevBot creating Beads epics directly | mitigate | SOUL.md rule: "NEVER create Beads epics — only Task Orchestrator creates epics"; enforced by directive, not by technical gate in Phase 8 |
| T-08-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan |
