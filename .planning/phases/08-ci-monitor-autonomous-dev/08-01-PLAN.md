Create the scripts directory and two files. Follow cc-openclaw convention: stdout = JSON only, stderr = human-readable logs, zsh strict mode, explicit binary paths.

scripts/lib/json-response.sh — Shared helper library (same boilerplate as email-triage agent):
- `ok_response()` function: prints `{"ok": true, "data": $1}`
- `error_response()` function: prints `{"ok": false, "error": "$1"}` and exits 1

scripts/poll-ci.sh — Core polling script per D-81, D-82, D-83, and the RESEARCH.md Pattern 1 reference implementation. Requirements:
- Shebang: `#!/usr/bin/env zsh` + `set -euo pipefail` (CLAUDE.md mandate)
- Constants: `AGENT_DIR="$HOME/.openclaw/agents/ci-monitor"`, `STATE_FILE="$AGENT_DIR/state/last-seen-runs.json"`, `REPOS_FILE="$AGENT_DIR/state/tracked-repos.txt"`, `GH="/opt/homebrew/bin/gh"`, `OC="/opt/homebrew/bin/openclaw"`
- Error on missing repos file: if `$REPOS_FILE` does not exist, write `{"ok": false, "error": "tracked-repos.txt not found"}` to stdout and exit 1
- Initialize state file: `[[ -f "$STATE_FILE" ]] || echo '{}' > "$STATE_FILE"`
- Main loop: read `$REPOS_FILE` line by line; skip blank lines and comment lines (`[[ -z "$repo" || "$repo" == \#* ]] && continue`)
- Per-repo: call `$GH run list -R "$repo" --status failure --json databaseId,conclusion,url,workflowName,headBranch,createdAt --limit 10 2>/dev/null || failures='[]'`
- Deduplication: use python3 to read `$STATE_FILE`, extract seen IDs for the repo, compute diff of new failures (current - seen)
- Per new failure: call `$GH run view "$run_id" -R "$repo" --json jobs 2>/dev/null` and use python3 to extract the first step with `conclusion == "failure"` (fall back to "unknown step")
- Alert: `PATH="/opt/homebrew/opt/node@24/bin:$PATH" "$OC" message send --channel telegram --target "$OPENCLAW_ANUJ_CHAT_ID" --message "CI FAILED [$repo] $workflow on $branch — step: $step — $run_url" 2>/dev/null || true`
  - The `|| true` means an alert failure does NOT abort the polling loop for other repos
- State update: after processing each repo, update `$STATE_FILE` with ALL current failure run IDs for that repo (not just new ones) so next poll deduplicates correctly
- Stdout output (final line): `{"ok": true, "new_failures": N, "alerted": BOOL}` where N is total count of new failures alerted across all repos and BOOL is true if N > 0
- Make the script executable: `chmod +x scripts/poll-ci.sh`

Do NOT embed the literal python3 code snippets from the research doc as-is — they use triple-quote shell variable interpolation that is fragile. Instead, write the python3 calls using temp files or clean heredoc patterns where variable interpolation is safe. Use `python3 -c` with single-argument strings and pass repo name, file path, etc. as positional arguments to python3 scripts or write a temp JSON file for input.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub API → poll-ci.sh | gh CLI fetches CI run data; the response is untrusted JSON parsed by python3 |
| poll-ci.sh → openclaw message send | Alert message is constructed from gh API fields (workflowName, headBranch, step name, url) — these are untrusted strings |
| Keychain → gateway env → OPENCLAW_ANUJ_CHAT_ID | Chat ID is injected via env var; must not appear in stdout/logs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Information Disclosure | poll-ci.sh stdout | mitigate | stdout is JSON only; `OPENCLAW_ANUJ_CHAT_ID` is never echoed; all debug goes to stderr |
| T-08-02 | Tampering | GitHub CI run data → shell string | mitigate | Workflow name, branch, and step name extracted by python3 and stored in zsh variables with explicit quoting; never interpolated raw into eval |
| T-08-03 | Denial of Service | gh run list API calls | mitigate | `--limit 10` caps results per repo per poll; 15 repos × 15 calls/hr = 225 calls/hr << 5,000 rate limit |
| T-08-04 | Information Disclosure | OPENCLAW_ANUJ_CHAT_ID in git history | mitigate | Stored in Keychain only; three secrets pipeline files source from `security find-generic-password` and are gitignored or contain only env-var references, not the raw value |
| T-08-SC | Tampering | npm/pip/cargo installs | accept | Phase 8 installs no new packages; all tools (gh, python3, openclaw) are from prior phases |
</threat_model>
