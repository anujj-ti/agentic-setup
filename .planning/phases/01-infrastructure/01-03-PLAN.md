Frozen naming convention (CLAUDE.md "Constraints → Secrets", RESEARCH.md Pattern 3):
- Keychain service: openclaw.<name> (lowercase, hyphens)
- Env var: OPENCLAW_<NAME> (uppercase, underscores)
- For this test: service `openclaw.test-secret`, env var `OPENCLAW_TEST_SECRET`

Three-file propagation (RESEARCH.md Pattern 3):
1. .openclaw/scripts/openclaw-secrets.sh — sourced by launchd at daemon start
2. .openclaw/scripts/openclaw-env.sh — sourced by interactive shells
3. secrets.sh (repo root, NOT stowed) — disaster-recovery provisioning array entry

Runtime-fetch pattern (NEVER hardcode value):
  export OPENCLAW_TEST_SECRET="$(security find-generic-password -s 'openclaw.test-secret' -w)"

Provisioning entry format (secrets.sh SECRETS array):
  "openclaw.test-secret|OPENCLAW_TEST_SECRET|<human-readable description>"

CLAUDE.md "Never echo secrets in terminal":
- Do NOT pass the value as a CLI argument to `security add-generic-password` (it appears in shell history and process listings).
- Two safe options:
  (a) Interactive prompt: `security add-generic-password -s "openclaw.test-secret" -a "$USER" -U -w` (no value arg — security reads from terminal).
  (b) Non-interactive stdin: `printf '%s' "<value>" | security add-generic-password -s "openclaw.test-secret" -a "$USER" -U -w` — the value reaches `security` via a pipe instead of argv, so it does not appear in shell history. Use this form when automation requires non-interactive entry.
- For this plan, the test value `phase1-test-value` is non-sensitive (it is a sentinel string for INFRA-03 verification, not a real credential), so option (b) is acceptable. The acceptance criterion below verifies the Keychain entry EXISTS and can be retrieved — it does NOT assert a specific value, so the operator may also choose option (a) and enter any non-empty string.

The /openclaw-add-secret skill (cc-openclaw/.claude/skills/openclaw-add-secret/SKILL.md) is now usable because Plan 01-04 established the OPENCLAW_REPO detection chain. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Keychain → repo files | Secret VALUES live only in Keychain; the three pipeline files contain only Keychain SERVICE NAMES and env var NAMES. The boundary is what makes git commits safe. |
| launchd shell sourcing | When the gateway daemon starts, launchd sources openclaw-secrets.sh which calls `security find-generic-password` at runtime — secrets enter the process environment but never the filesystem |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | Plaintext secret values in repo files | mitigate | The runtime-fetch pattern stores `"$(security find-generic-password -s 'openclaw.test-secret' -w)"` — the COMMAND that fetches the value at runtime — never the value itself. Acceptance criterion explicitly asserts the value `phase1-test-value` does NOT appear in any file. |
| T-03-02 | Information Disclosure | Secrets in shell history / argv | mitigate | Per CLAUDE.md "Never echo secrets in terminal", the action forbids passing the value as a `-w <value>` argument to `security add-generic-password`. Acceptable forms: (a) interactive prompt (`-w` with no value), or (b) stdin pipe `printf ... | security ... -w` — both keep the value out of argv. For this test the value is a sentinel string (not a real credential), but the procedure mirrors what Phase 2 will use for the Telegram token. |
| T-03-03 | Information Disclosure | secrets.sh committed to git | accept | secrets.sh contains only service names, env var names, and human-readable descriptions — by design, no values. CLAUDE.md "Secrets" rule explicitly permits this provisioning script in the repo. |
| T-03-04 | Tampering | Concurrent writes to the three files | mitigate | The cc-openclaw skill is the single writer (D — skill-only edits). The action explicitly forbids manual file edits in parallel with the skill invocation. Idempotency check prevents duplicate appends on re-run. |
| T-03-05 | Repudiation | Untracked secret additions | accept | secrets.sh is git-tracked; every addition appears in commit history. Keychain modifications by the cc-openclaw skill are observable via the file change. No additional logging required for v1. |
</threat_model>
