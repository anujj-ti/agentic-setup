Shell convention (CLAUDE.md "Shell Scripting Conventions"):
- Shebang: #!/usr/bin/env zsh (NEVER #!/bin/bash — macOS bash is 3.2)
- Strict mode: set -euo pipefail
- Output protocol: stdout = JSON only, stderr = human logs (via print "..." >&2)
- JSON shape: {"ok":true,"data":{...}} or {"ok":false,"error":"snake_case_code"}

Secrets naming (CLAUDE.md "Constraints → Secrets"):
- Keychain service: openclaw.<name> (lowercase, hyphens)
- Env var: OPENCLAW_<NAME> (uppercase, underscores)

Architecture detection (RESEARCH.md Pitfall 5):
- $(uname -m) returns "arm64" on Apple Silicon, "x86_64" on Intel
- node@24 keg-only paths:
  - Apple Silicon: /opt/homebrew/opt/node@24/bin
  - Intel: /usr/local/opt/node@24/bin

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Network → host (curl installer) | OpenClaw install script downloaded over HTTPS from openclaw.ai is executed with the user's shell privileges |
| Homebrew → host | brew install runs formula-supplied install scripts (node@24, stow, jq) |
| launchd → repo files | The ai.openclaw.gateway LaunchAgent will eventually source files inside the repo (.openclaw/scripts/openclaw-secrets.sh) — those files become attack surface for any future repo write |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | curl-piped install of openclaw.ai/install.sh | accept | Recommended path per docs.openclaw.ai/install. openclaw npm package is well-established (894k weekly downloads, verified slopcheck OK per Package Legitimacy Audit). HTTPS guarantees server identity. The installer is run interactively by the user, who can inspect script behavior before piping to bash if desired. |
| T-01-02 | Tampering | Homebrew formula installs (node@24, stow, jq) | accept | Official Homebrew formulae for upstream-published tools (Node.js, GNU Stow, jq). No third-party taps. |
| T-01-03 | Information Disclosure | secrets.sh in repo root | mitigate | secrets.sh contains only Keychain SERVICE NAMES and ENV VAR NAMES — never actual secret values. CLAUDE.md "Constraints → Secrets" forbids plaintext secrets in tracked files. The disaster-recovery script invokes `security add-generic-password -s "$service" -a "$USER" -w` (no value argument) which prompts the user interactively. |
| T-01-04 | Information Disclosure | .openclaw/scripts/openclaw-secrets.sh stub | mitigate | Stub contains no secrets — only header comments and a placeholder marker. Future secrets appended by /openclaw-add-secret use `security find-generic-password ... -w` to fetch values at runtime from Keychain (never hardcoded). |
| T-01-05 | Tampering | nvm-managed Node visible to launchd | mitigate | RESEARCH.md Pitfall 2: launchd does not source shell profiles, so nvm Node is invisible to the daemon. install-prereqs.sh pins the Homebrew node@24 absolute path; the conditional append into openclaw-secrets.sh ensures launchd sees node@24 once Plan 01-04 wires stow. Mitigation matures in Plan 01-04. |
| T-01-06 | Elevation of Privilege | Permissions on ~/.openclaw after first stow | accept (this plan) | This plan does not stow yet. Plan 01-04 handles permissions. Recorded here for tracking. |
| T-01-SC | Tampering | npm/brew installs (openclaw, node@24, stow, jq) | mitigate | RESEARCH.md "Package Legitimacy Audit": openclaw (npm, 894k weekly downloads, slopcheck OK), node@24 (official Homebrew formula, N/A), stow (official Homebrew formula for GNU tool, N/A), jq (industry standard, N/A). No [ASSUMED] or [SUS] packages — no blocking checkpoint required. |
</threat_model>
