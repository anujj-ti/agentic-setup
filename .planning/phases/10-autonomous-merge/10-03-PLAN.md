Critical squash-vs-merge distinction (from D-101 and research open questions):
- D-101: use gh pr merge --squash
- Squash merge creates a SINGLE regular commit (not a merge commit) on the base branch
- Plain git commits have only ONE parent, so git revert <sha> --no-edit works directly
- The -m 1 flag is ONLY needed for true merge commits (with two parents)
- D-103: revert = gh pr reopen + git revert <sha> + new Notion log entry (no -m flag)
- Do NOT use git revert -m 1 for squash merge reverts

Pitfall from research to avoid:
- gh pr reopen succeeds even when --delete-branch was used; head branch is not recreated
- Document this as a comment in the revert script output JSON

Create devbot-revert-merge.sh with #!/usr/bin/env zsh and set -euo pipefail.

Positional arguments:
  MERGE_SHA=$1         — the squash merge commit SHA (from Notion page / devbot-merge-pr.sh output)
  PR_NUMBER=$2         — the original PR number (integer)
  ORIGINAL_PAGE_ID=$3  — the Notion page ID of the original merge decision log entry

Validate all three args are present and non-empty. If any is missing, print {"ok":false,"error":"usage: devbot-revert-merge.sh <merge_sha> <pr_number> <original_page_id>"} and exit 1.

Step 1 — Create revert commit: run git revert "$MERGE_SHA" --no-edit
  Note: No -m flag. Per D-101 and D-103, this is a squash merge commit (single parent). git revert on a single-parent commit does not need -m.
  On failure: print {"ok":false,"error":"git revert failed — SHA may be incorrect or working tree is dirty"} and exit 1.

Step 2 — Push revert commit: run git push origin HEAD
  On failure: print {"ok":false,"error":"git push failed"} and exit 1.

Step 3 — Reopen PR: run /opt/homebrew/bin/gh pr reopen "$PR_NUMBER" --comment "Merge reverted. Squash commit: ${MERGE_SHA}. See original Notion decision: ${ORIGINAL_PAGE_ID}. Note: head branch was deleted at merge time — recreate branch to push new commits."
  On failure: log to stderr "gh pr reopen failed — PR may already be open or merged into another branch" but continue (non-fatal, revert commit already exists).

Step 4 — Log revert to Notion: run node scripts/notion-log-decision.js with args:
  --action "revert merge PR #${PR_NUMBER}"
  --rationale "User requested revert via Notion decision log"
  --reversibility "permanent — creates new revert commit ${MERGE_SHA}-revert; cannot be un-reverted automatically"
  --evidence "originalPageId:${ORIGINAL_PAGE_ID}, mergeCommitSha:${MERGE_SHA}"
  Capture as REVERT_PAGE_ID. If empty or node exits non-zero (set -e catches), log to stderr "Notion revert log failed" but do NOT exit — the revert commit is already pushed and is the source of truth.

Output on completion: {"ok":true,"revertedSha":"${MERGE_SHA}","prNumber":"${PR_NUMBER}","originalPageId":"${ORIGINAL_PAGE_ID}","revertNotionPageId":"${REVERT_PAGE_ID:-not-logged}"}

chmod +x the script.
